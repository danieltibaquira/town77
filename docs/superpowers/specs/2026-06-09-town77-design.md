# Town 77 — Design Spec

**Date:** 2026-06-09  
**Status:** Approved  
**Scope:** Full-stack multiplayer browser game — engine, server, client, theming, testing

---

## 1. Game Overview

Town 77 is a turn-based, multiplayer tile-placement strategy game for 2–5 players. Players place chips on a 7×7 grid following color/shape constraints, draw replacements from a shared bag, and compete across two independent scoring criteria. All game parameters (grid size, chip composition, theme) are configurable before room creation.

---

## 2. Rules

### 2.1 Components

- **Grid**: 7×7 (configurable). 49 cells.
- **Chips**: Configurable composition. Default: 7 colors × 7 shapes × 1 copy = 49 unique chips.
- **Hand**: Each player holds 4 chips (configurable).
- **Bag**: All chips not in hands, shuffled. Source for draws.

### 2.2 Setup

1. All chips shuffled into the bag using a seeded RNG (seed stored in room, enables session recovery).
2. Each player dealt `handSize` chips from the bag.
3. First player selected randomly.

### 2.3 Placement Rules

A chip placement is valid if and only if:

1. **Adjacency**: The target cell shares a side (not diagonal) with at least one already-occupied cell. Exception: the very first chip of the game may be placed on any empty cell.
2. **Row uniqueness**: No chip in the same row shares the placed chip's `color`. No chip in the same row shares the placed chip's `shape`.
3. **Column uniqueness**: No chip in the same column shares the placed chip's `color`. No chip in the same column shares the placed chip's `shape`.

### 2.4 Turn Actions

On each turn a player must choose exactly one of:

| Action | Description | Cost |
|--------|-------------|------|
| **Place** | Place 1 valid chip from hand onto grid. Draw 1 from bag (if available). | — |
| **Exchange** | Return 3–4 hand chips sharing the same color OR same shape back to bag. Bag shuffled. Draw same count immediately. Net zero hand change. | Turn consumed |
| **Discard** | Discard 1 chip permanently. Draw 1 from bag (if available). | Once per game per player |

### 2.5 Game End

The game ends when the bag is empty and no player can place a valid chip.

### 2.6 Scoring

Two independent criteria, both calculated at game end:

| Criterion | Calculation | Direction |
|-----------|-------------|-----------|
| **Fichas colocadas** | Total chips successfully placed on grid | Higher = better |
| **Fichas restantes** | Chips remaining in hand at game end | Lower = better |

**Combined score**: `(placed × placedWeight) - (remaining × remainingWeight)` — weights configurable in `GameConfig`, default both `1`.

Results screen announces both independent leaders and the combined score winner.

## 3. Architecture

### 3.1 Monorepo Structure

```
town77/
├── packages/
│   ├── shared-types/       # TS interfaces — Chip, GameConfig, Theme, socket events
│   ├── game-engine/        # Pure TS — grid, validation, bag, scoring, turn logic
│   ├── server/             # Express + Socket.IO + SQLite persistence
│   ├── client/             # React 18 + Vite + Framer Motion + Zustand
│   └── e2e/                # Playwright end-to-end tests
├── docs/
├── docker-compose.yml
├── docker-compose.dev.yml
├── nginx/
│   └── nginx.conf          # Reverse proxy, WebSocket upgrade headers
├── pnpm-workspace.yaml
└── biome.json
```

### 3.2 Data Flow

```
Client drag-drop
  → game-engine.isValidPlacement() [client-side, instant cell highlighting]
  → socket.emit('place_chip', { chip, row, col })
  → server: game-engine.isValidPlacement() [authoritative re-validation]
  → server: applyPlacement() → persist to SQLite
  → socket.broadcast('state_update', fullState)
  → all clients: animate transition via Framer Motion
```

Server is the single source of truth. Client validation is UX-only (highlight valid cells, prevent obviously invalid drags). Server always re-validates.

### 3.3 Package Responsibilities

**`shared-types`**: Pure TypeScript interfaces and enums. No runtime logic. Imported by all other packages. Never has side effects.

**`game-engine`**: Pure functions only. No I/O. No sockets. No DOM. Accepts injectable `RNG`. Exports: `validatePlacement`, `getValidCells`, `initBag`, `dealHands`, `applyPlacement`, `doExchange`, `doDiscard`, `isGameOver`, `calculateScores`.

**`server`**: Orchestrates rooms and turns. Owns socket event handlers, room lifecycle, SQLite reads/writes. Calls `game-engine` for all rule checks.

**`client`**: Renders game state received from server. Calls `game-engine` only for local UX feedback (valid cell highlighting). Manages local UI state via Zustand. All user-facing strings via `react-i18next`.

**`e2e`**: Playwright tests only. No game logic. Tests real WebSocket flows via browser automation.

## 4. Data Model

### 4.1 TypeScript Types (`shared-types`)

```typescript
type ColorId = string   // 'color-1' … 'color-N'
type ShapeId = string   // 'cottage', 'tower', 'rowhouse', etc.

interface Chip {
  color: ColorId
  shape: ShapeId
}

interface GameConfig {
  grid:     { rows: number; cols: number }
  chips:    { colors: ColorId[]; shapes: ShapeId[]; copies: number }
  handSize: number
  scoring:  { placedWeight: number; remainingWeight: number }
  exchange: { min: number; max: number }  // default 3–4
}

interface PlayerState {
  id:           string
  name:         string
  hand:         Chip[]
  placed:       number
  hasDiscarded: boolean
  connected:    boolean
}

interface GameState {
  grid:      (Chip | null)[][]
  bag:       Chip[]
  players:   PlayerState[]
  turnIndex: number
  phase:     'lobby' | 'playing' | 'finished'
  config:    GameConfig
  themeId:   string
  seed:      number
}

interface Score {
  playerId:  string
  name:      string
  placed:    number
  remaining: number
  combined:  number
}
```

### 4.2 SQLite Schema

```sql
CREATE TABLE rooms (
  code        TEXT PRIMARY KEY,
  theme_id    TEXT NOT NULL,
  config_json TEXT NOT NULL,
  state_json  TEXT NOT NULL,
  seed        INTEGER NOT NULL,
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL
);

CREATE TABLE players (
  id            TEXT PRIMARY KEY,
  room_code     TEXT NOT NULL REFERENCES rooms(code),
  name          TEXT NOT NULL,
  session_token TEXT NOT NULL UNIQUE,
  created_at    INTEGER NOT NULL
);

CREATE INDEX idx_players_room ON players(room_code);
```

`state_json` is the full serialized `GameState`. Written on every state mutation. Rooms never deleted automatically — TTL policy handled by a background cleanup job (configurable, default 30 days since `updated_at`).

### 4.3 Socket Events

**Client → Server**

| Event | Payload |
|-------|---------|
| `create_room` | `{ config: GameConfig, themeId: string, playerName: string }` |
| `join_room` | `{ code: string, playerName: string, playerId?: string }` |
| `start_game` | `{}` (host only) |
| `place_chip` | `{ chip: Chip, row: number, col: number }` |
| `exchange_chips` | `{ chips: Chip[] }` |
| `discard_chip` | `{ chip: Chip }` |

**Server → Client**

| Event | Payload |
|-------|---------|
| `room_joined` | `{ code: string, playerId: string, sessionToken: string, state: GameState }` |
| `state_update` | `{ state: GameState }` |
| `error` | `{ code: string, messageKey: string }` |
| `game_over` | `{ scores: Score[] }` |

interface AnimationPreset {

  shapes:          Record<ShapeId, string>      // ShapeId → SVG path string

### 5.2 Town 77 Default Theme

**Color palette** (town materials):

| Token | Hex | Name |
|-------|-----|------|
| `color-1` | `#B04A2F` | Terracotta |
| `color-2` | `#3D7AB5` | Slate Blue |
| `color-3` | `#4A7C59` | Forest Green |
| `color-4` | `#C4A35A` | Sandstone |
| `color-5` | `#8B3A52` | Burgundy |
| `color-6` | `#2B2F5E` | Midnight |
| `color-7` | `#E8D5B0` | Cream/Gold |

**Surfaces**: near-black with warm undertone (`#0F0D17` bg, `#1C1828` grid, `#241F35` cell).

**Shapes** (house silhouettes, SVG paths): `cottage`, `rowhouse`, `tower`, `victorian`, `barn`, `bungalow`, `skyscraper`.

**Fonts**: Bebas Neue (display), Inter (UI).

**Animation preset**: `flashy` — spring stiffness 260, damping 20, particle burst on placement, shimmer wave on row/column completion.

### 5.3 Theme Loading

Themes are JSON files loaded at runtime from `public/themes/{id}.json`. The client injects CSS custom properties on theme change — zero component re-renders for token changes. Framer Motion variants receive the `animationPreset` via React context (`ThemeContext`).

Swapping theme = load JSON → inject CSS vars → update context. No logic changes anywhere.

## 7. Screens & User Flow

### 7.1 Flow

```
Home
  ├── [Jugar] → Config Screen → Create Room → Lobby → Game → Results
  └── [Unirse] → Enter Code → Lobby → Game → Results
```

### 7.2 Screen Descriptions

**Home**: Two CTAs only — `Jugar` and `Unirse`. Full-bleed theme animation in background.

**Config Screen** (host, pre-room):
- Theme picker: card grid with live miniature preview of chip shapes + palette
- Quick presets: `Clásico 7×7`, `Rápido 5×5`, `Personalizado`
- Grid size: discrete button group (`5×5`, `7×7`, `9×9`, `11×11`)
- Colors: stepper with live swatch preview
- Shapes: stepper with live shape icon preview
- Copies per combo: stepper (`1`, `2`, `3`)
- Hand size: stepper (`3`–`6`)
- Derived display: `Total fichas: N` and `Celdas del tablero: M` — warns if chips < cells
- CTA: `Crear sala` → generates room code

**Lobby**: Room code displayed large + copy + share link. Player list with connection indicator. Config summary (read-only). Host sees `Iniciar juego` when ≥2 players.

**Game Screen**:
- Center: 7×7 (or configured) grid — chips as themed SVG silhouettes
- Bottom: player's hand — horizontal scrollable strip, drag-and-drop chips
- Top bar: all players' placed counts + connected status + whose turn indicator
- Sidebar/overlay: bag count, discard/exchange action buttons (grayed when unavailable)
- Turn indicator: animated swoosh with current player name

**Results Screen**:
- Full score table: `Colocadas | Restantes | Total`
- Winner crown(s) — separate icons for each criterion + combined
- `Jugar de nuevo` and `Nueva sala` buttons

### 7.3 Session Recovery

Player can recover any game session by navigating to `/room/{code}` and entering their name. Server matches by `sessionToken` stored in `localStorage`. If token matches a player in the room, full state is restored immediately.

### 9.2 Deterministic Validation — Injectable RNG

All randomness in `game-engine` flows through an injected `RNG` interface. `Math.random()` never called directly.

```typescript
export interface RNG { nextFloat(): number }

export class MathRNG implements RNG {
  nextFloat() { return Math.random() }
}

export class SeededRNG implements RNG {
  constructor(private rng: Prando) {}
  nextFloat() { return this.rng.next() }
}
```

Every function that needs randomness (`initBag`, `dealHands`, `doExchange`, `selectFirstPlayer`) accepts `rng: RNG` as a parameter.

### 9.3 Property-Based Tests (`@fast-check/vitest`)

```typescript
// Invariant: any valid placement preserves grid constraints
test.prop([fc.integer(0,6), fc.integer(0,6), arbitraryChip()])(
  'placement preserves row/column uniqueness',
  (row, col, chip) => {
    fc.pre(isValidPlacement(grid, row, col, chip))
    const next = applyPlacement(grid, row, col, chip)
    expect(gridIsConsistent(next)).toBe(true)
  }
)

// Invariant: exchange always preserves hand size
test.prop([arbitraryHand(), arbitraryBag()])(
  'exchange preserves hand count',
  (hand, bag) => {
    fc.pre(canExchange(hand))
    const { newHand } = doExchange(hand, bag, seededRng)
    expect(newHand.length).toBe(hand.length)
  }
)

// Invariant: scores are always non-negative
test.prop([arbitraryPlayerState()])(
  'scores are non-negative',
  (player) => {
    const score = calculateScore(player, defaultConfig.scoring)
    expect(score.placed).toBeGreaterThanOrEqual(0)
    expect(score.remaining).toBeGreaterThanOrEqual(0)
  }
)
```

## 10. Error Handling

Server validates every action via `game-engine` before applying. On violation → `error` event with `{ code, messageKey }` → client looks up `messageKey` in `errors.json` → inline toast, no state change.

**Error codes**: `INVALID_PLACEMENT`, `NOT_YOUR_TURN`, `EXCHANGE_RULE_VIOLATED`, `ALREADY_DISCARDED`, `ROOM_NOT_FOUND`, `GAME_ALREADY_STARTED`, `INVALID_CHIP`.

**Disconnect**: Player slot held indefinitely. `connected: false` shown in UI to other players. When a disconnected player's turn arrives, a countdown timer starts (configurable, default 30s). If they do not reconnect within the window, their turn is automatically skipped (no action taken, turn passes to next player). The slot and game state are preserved — they may reconnect at any time and resume. Game never ends due to disconnection alone.

**Session recovery**: `GET /room/:code` + sessionToken from localStorage → server rehydrates full state from SQLite → `room_joined` event with full state.
